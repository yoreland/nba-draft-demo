# Weighted consensus draft simulator with scraper-backed data pipeline

A Python CLI tool that scrapes NBA mock draft data from four sources (NBADraft.net, Tankathon, ESPN, betting odds sites), normalizes player names with fuzzy matching, and produces a weighted consensus draft board. The aggregation engine weights each source by authority (betting odds highest) and recency (exponential time decay). A backtest mode validates the approach against the actual 2024 draft, achieving 33% exact match and 77% within-3-picks accuracy on hardcoded historical predictions. A fallback path provides static 2026 prospect data when scraping fails.

**Watch for:** The prediction output is dominated by whichever scrapers happen to succeed — there's no minimum-source threshold, so a run where only one low-quality source responds still produces a full 30-pick board with no warning that confidence is degraded (confirmed). The fallback data in `main.py` references a different 2026 prospect list than what the actual scrapers return, suggesting the fallback is stale or speculative (confirmed). The fuzzy matcher has a quadratic scan over all existing player groups on every insertion, which is fine for 30 picks × 4 sources but would break silently at scale (confirmed, acceptable for demo scope).

## High-level view

The aggregation engine is the only novel logic in the project. It groups predictions by fuzzy-matched player name, computes a weighted average pick position using `source_weight × time_decay × confidence`, and assigns pick slots by sorting the averages. The approach is sound for a demo — it naturally surfaces consensus — but the single-source scenario exposes a gap where the output looks authoritative despite thin data backing.

The scraper layer is four independent modules that each try to parse one site's HTML structure. They're written defensively (broad exception catches, empty-list returns on failure) and the orchestrator in `scrapers/__init__.py` runs them sequentially. When live scraping fails in a sandboxed or rate-limited environment, execution silently falls through to the hardcoded fallback in `main.py`, which produces a reasonable-looking board from static data that may diverge from reality over time.

The backtest validates the aggregation logic against a 2024 ground-truth JSON file, but only exercises the aggregation engine — not the scrapers. The hardcoded mock predictions used for backtesting are already very close to the actual draft order (they represent real pre-draft consensus), so the 77% within-3 number validates that the weighted average doesn't distort good input, rather than proving the system can generate good predictions from noisy data.

The fuzzy name matching handles suffixes, accents, and punctuation, but the `_similarity_ratio` function uses a custom metric that diverges from standard edit-distance semantics. For the 30-player scale this works, but it could produce false positives on names with high character overlap (e.g., "Cameron Carr" vs "Cameron Carter").

1. **No minimum-source gate** — When only one scraper succeeds, the output carries no indication that consensus confidence is degraded. Add a minimum-source count or surface a warning/lower confidence scores when fewer than N sources contribute.
2. **Stale fallback prospect list** — The fallback 2026 data in `main.py` differs from what live scrapers return. It should be updated to reflect the most recent consensus before draft day, or generated from a cached successful scrape rather than hand-coded.
3. **Backtest doesn't validate scraper reliability** — The backtest uses hardcoded predictions, so it cannot catch regressions in HTML parsing or site structure changes. A lightweight scraper integration test (even against saved HTML fixtures) would catch this.
4. **Fuzzy match false-positive risk on similar names** — The custom `_similarity_ratio` doesn't penalize transpositions or insertions, meaning players with overlapping character sets (same first name, similar last name) could merge incorrectly.
5. **Time decay is no-op in backtest** — All four hardcoded 2024 sources use dates within 12 days of each other, so `exp(-0.01 * days)` produces nearly identical decay factors (~0.998 vs ~0.988). The time-decay mechanism is never meaningfully exercised by tests.

## Aggregation engine design and the single-source problem

The weighted consensus formula `weighted_pick = Σ(pick × source_weight × time_decay × confidence) / Σ(weights)` produces a position-weighted average that naturally favors higher-authority, more-recent sources. This is a good fit for the problem: betting odds move closer to reality as the draft approaches, so time decay and the 0.95 weight for odds creates a system that increasingly trusts the market.

The gap is what happens when the input is thin. `aggregate_predictions` in `aggregator.py` builds and returns a full 30-pick board regardless of whether one source or four contributed. The `consensus_score` field (computed as `total_weight / len(preds)`) does decrease with fewer sources, but this isn't surfaced to the user as a quality signal. In the actual output file, picks backed by a single source show `consensus_score: 0.4858` while multi-source picks show `0.6813`, but there's no threshold below which the system says "confidence too low to rank this player."

For a competition where accuracy matters, this means a flaky network day could silently produce a board dominated by one source's biases.

## Scraper fragility and the fallback trap

Each scraper targets specific CSS selectors (`.mock-row`, `td.rank`, `td.player`, etc.) that will break when the source sites redesign. This is expected and acceptable for a demo, but the failure mode is silent: the scraper returns an empty list, the orchestrator logs a warning, and execution continues.

The fallback in `_get_fallback_2026_predictions()` contains 30 ESPN-attributed picks plus 10 betting-odds picks — 40 predictions total. This data was clearly written at a point in time (it lists Cooper Flagg as consensus #1, which matches early-cycle predictions) but the actual scraper output already shows a different #1 (AJ Dybantsa from live scraping). The fallback will produce increasingly wrong results as the draft approaches unless manually updated.

A better pattern: cache the last successful scrape to disk and use that as fallback, so the "stale" data is at least the most recent real data rather than hand-coded speculation.

## Fuzzy matching — works for now, edge cases ahead

The name normalization pipeline handles the common cases well: `"DaRon Holmes II"` → `"daron holmes"`, `"Kel'el Ware"` → `"kelel ware"`. The fuzzy matcher layers three strategies — normalized exact match, last-name + first-initial match, and a character-level similarity ratio.

The custom `_similarity_ratio` computes `(positional_matches/max_len * 0.6) + (set_intersection/set_union * 0.4)`. This is not Levenshtein or Jaro-Winkler — it's biased toward strings that share many characters regardless of order. For a name like "Cameron Carr" vs "Cameron Carter", the set overlap is high (all of "cameroncr" appears in both) and the positional match is high (first 10 characters match). The threshold is 0.92 for short strings, which may not be enough to separate these.

In the current 30-player-per-source dataset, this hasn't caused problems. But for a 2026 draft board where multiple "Cameron" or "Bailey" names appear (Ace Bailey, Airious Bailey both appear in the fallback), the risk is non-trivial. The actual output shows these players correctly separated, but the separation depends on the short-string threshold holding.

## Backtest validity — measuring the right thing?

The backtest reports 33% exact match and 77% within-3 on 30 picks. These numbers validate that the aggregation engine preserves consensus rather than distorting it. But the input data is already near-perfect: the four hardcoded 2024 sources differ from the actual draft by at most a few positions in the top 5 and somewhat more in picks 15-30. A straight average of any two of these sources would likely achieve similar accuracy.

What the backtest doesn't test:
- Whether the scrapers can actually extract data from live sites
- Whether the fuzzy matcher correctly merges names across sources in adversarial conditions
- Whether the time decay provides meaningful signal when source dates span weeks rather than days
- Whether the system degrades gracefully when one source is wildly wrong

The test assertions in `test_backtest.py` use low bars (`within_3_pct > 30`) that would pass even with a random shuffling algorithm on near-correct input.

## Test coverage shape

The test suite covers models (serialization round-trips), aggregator logic (single-source, multi-source, disagreement, name merging), and backtest execution. What's absent:

- No tests for any scraper module (parsing logic is entirely untested)
- No tests for the fallback path in `main.py`
- No tests for `_odds_to_confidence` in the odds scraper
- No tests that the fuzzy matcher correctly rejects similar-but-different names at the boundary
- No integration test that runs `main.py` end-to-end

The scraper code is the most fragile part of the system (dependent on external HTML structure) and the least tested.
